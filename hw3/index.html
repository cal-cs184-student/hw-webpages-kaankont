Overview

In this project we built a physically-based path tracer. We started by generating camera rays and testing them against triangles and spheres, then added a BVH so we don't have to check every primitive for every ray. After that we implemented two direct lighting strategies (hemisphere sampling and importance sampling over lights), then recursive indirect lighting with Russian Roulette for global illumination, and finally adaptive sampling to avoid wasting samples on pixels that already converged. The coolest part was seeing color bleeding show up for the first time after getting indirect lighting working.


Part 1: Ray Generation and Scene Intersection (20 Points)

Walk through the ray generation and primitive intersection parts of the rendering pipeline.

The rendering pipeline starts at raytrace_pixel, which takes a pixel coordinate (x, y) and shoots multiple rays through it. For each sample, we pick a random point within the pixel using a uniform grid sampler, then normalize those coordinates into [0,1] range by dividing by the image width and height. These normalized coordinates get passed to Camera::generate_ray.

In generate_ray, we map the [0,1] coordinates onto the camera's sensor plane. The sensor plane sits at z = -1 in camera space, and its extent is determined by the field of view. Specifically, the x-coordinate maps to [-tan(hFov/2), tan(hFov/2)] and y maps to [-tan(vFov/2), tan(vFov/2)]. We construct a direction vector from the origin through this sensor point, normalize it, then transform it from camera space to world space using the camera-to-world rotation matrix c2w. The resulting ray starts at the camera position with min_t = nClip and max_t = fClip.

Each ray then gets tested against scene primitives. When a ray hits something, we record the closest intersection by updating ray.max_t, which prevents later intersections from overwriting nearer ones. The intersection stores the hit time t, the surface normal (interpolated for triangles), a pointer to the primitive, and its BSDF material.


Explain the triangle intersection algorithm you implemented in your own words.

We used the Moller-Trumbore algorithm. The idea is to solve for the ray parameter t and barycentric coordinates (b1, b2) simultaneously by setting up a linear system. Given triangle vertices p1, p2, p3 and a ray with origin o and direction d, We compute edge vectors e1 = p2 - p1 and e2 = p3 - p1, along with helper vectors s = o - p1, s1 = cross(d, e2), and s2 = cross(s, e1).

The denominator is dot(s1, e1). If this is close to zero the ray is parallel to the triangle so we bail out. Otherwise we compute t = dot(s2, e2) / denom, b1 = dot(s1, s) / denom, and b2 = dot(s2, d) / denom, with b0 = 1 - b1 - b2. If any barycentric coordinate is negative the hit point is outside the triangle. If t falls outside [min_t, max_t] we also reject it. On a valid hit we update r.max_t = t and interpolate the surface normal as b0*n1 + b1*n2 + b2*n3 using the vertex normals.

For sphere intersection we used the standard quadratic formula approach. Substituting the ray equation into the sphere equation |p - center|^2 = r^2 gives a quadratic in t. We compute the discriminant, and if it's non-negative we get two solutions t1 and t2 (the entry and exit points). We pick the smallest one that falls within the valid t range.

Part 2: Bounding Volume Hierarchy (17 Points)

Walk through your BVH construction algorithm. Explain the heuristic you chose for picking the splitting point.

Our BVH construction is recursive. At each step, we first compute the bounding box of all primitives in the current range. If the number of primitives is at or below max_leaf_size, we make it a leaf node and return.

For internal nodes, we need to decide how to split the primitives into two groups. We compute a second bounding box over just the centroids of all the primitives, then pick the axis along which the centroids are most spread out (the longest extent of the centroid bounding box). The split point is the midpoint of the centroid box along that axis. We use std::partition to rearrange the primitives so that everything with a centroid below the split goes to the left child and everything else goes to the right.

There's one edge case to handle: if all centroids land on the same side of the split (which happens when many primitives share the same centroid position), both children would be empty on one side. We handle this by just splitting at the halfway index, which guarantees both children get at least one primitive.

Compare rendering times on a few scenes with moderately complex geometries with and without BVH acceleration. Present your results in a one-paragraph analysis.

With BVH acceleration, rendering complex scenes is extremely fast. CBbunny (about 33k triangles) rendered in roughly 1.6 seconds, dragon (about 100k triangles) in 1.8 seconds, and wall-e (about 240k triangles) in 1.4 seconds, all at 16 samples per pixel on 30 threads. Without BVH, each ray would need to be tested against every single primitive in the scene, meaning a scene with 100k triangles would require 100k intersection tests per ray. With BVH, the tree structure lets us skip huge chunks of the scene by testing against bounding boxes first. In practice this brings the average number of intersection tests down to something like 5-10 per ray regardless of scene complexity, which is why dragon (100k triangles) and CBbunny (33k triangles) render in about the same time. The speedup is roughly proportional to N / log(N), so for the dragon scene that is around a 10,000x improvement. Without BVH these scenes would take hours instead of seconds.


Part 3: Direct Illumination (20 Points)

Walk through both implementations of the direct lighting function.

Uniform hemisphere sampling: At each hit point we set up a local coordinate system with the surface normal aligned to the z-axis. Then we take num_lights * ns_area_light samples uniformly distributed over the hemisphere. For each sample direction, we shoot a ray from the hit point and check if it hits anything. If it does, we grab the emission from whatever it hit (which will be zero for non-light surfaces and nonzero for lights). The contribution for each sample is f(w_out, w_in) * emission * cos(theta) divided by the hemisphere PDF of 1/(2*PI). We average over all samples.

Importance sampling (light sampling): Instead of sampling random directions and hoping to hit a light, we loop over each light in the scene and sample directions toward it directly. For each light, we take ns_area_light samples (or just 1 for point/directional lights since they're delta distributions). The light's sample_L function gives us a sampled direction, the distance to the light, the radiance, and the PDF. We transform the direction into local coordinates, check that it's above the surface (cos_theta > 0), then shoot a shadow ray to see if anything blocks the path. The shadow ray's max_t is set to distToLight - EPS_F so it doesn't accidentally intersect the light itself. If the path is clear, the contribution is radiance * f(w_out, w_in) * cos(theta) / pdf.

Part 4: Global Illumination (25 Points)

Walk through your implementation of the indirect lighting function.

The indirect lighting function at_least_one_bounce_radiance handles all light that has bounced at least once. At each hit point, if we're accumulating bounces (or if this is the last allowed bounce), we add the one-bounce direct lighting contribution. Then, if we still have bounce budget left (r.depth > 1), we sample a new direction from the BSDF and trace a recursive bounce ray.

To keep the computation unbiased but finite, we use Russian Roulette. The first indirect bounce always happens (so we don't lose energy at low depths), but after that each continuation has a 65% probability. When a ray does continue, we divide its contribution by the continuation probability (0.65) to compensate and keep the estimator unbiased. The full contribution of a bounce is indirect_radiance * bsdf * cos(theta) / pdf / continuation_probability.

The top-level function est_radiance_global_illumination handles two modes via the isAccumBounces flag. When accumulating, it adds zero-bounce (emission) plus all indirect bounces together for the full global illumination result. When not accumulating, it returns only the contribution from exactly the m-th bounce, which is useful for visualizing how each bounce contributes individually.

Part 5: Adaptive Sampling (15 Points)

Explain adaptive sampling. Walk through your implementation of the adaptive sampling.

The idea behind adaptive sampling is that not all pixels need the same number of samples. A flat wall with uniform lighting converges fast, but edges and shadow boundaries need way more. So instead of blasting every pixel with the same sample count, we check if a pixel has converged and stop early if it has.

Our implementation tracks running sums s1 (sum of illuminance values) and s2 (sum of squared illuminance values) as samples are taken. Every samplesPerBatch samples, we compute the sample mean and variance using these running sums: mean = s1/n and variance = (s2 - s1*s1/n) / (n-1). From the variance we get the standard deviation, and then we compute a 95% confidence interval half-width: I = 1.96 * stddev / sqrt(n). If I <= maxTolerance * mean, the pixel has converged and we stop sampling it. The actual number of samples taken per pixel gets stored in sampleCountBuffer so we can visualize the sampling rate.

Part 6: Extra Credit - Surface Area Heuristic for BVH

Approach

We replaced the naive centroid-midpoint BVH split with a Surface Area Heuristic (SAH) using a bucket-based approach. The idea is that instead of blindly splitting at the midpoint along the longest axis, we should try multiple candidate splits and pick the one that minimizes the expected cost of a ray traversing the tree. The SAH cost model says that the probability of a ray entering a child node is proportional to the surface area of that child's bounding box relative to the parent's. So a good split minimizes SA_left * N_left + SA_right * N_right, where SA is the surface area of each child's bounding box and N is the number of primitives on each side.

Implementation

We divide the chosen axis into 12 buckets. Each primitive gets assigned to a bucket based on where its centroid falls along that axis. Then for each of the 11 possible splits between adjacent buckets, we compute the SAH cost by accumulating the bounding box and count on each side. We pick the split with the lowest cost and partition the primitives accordingly. If all primitives end up in the same bucket (degenerate case), we fall back to splitting at the halfway index just like before.

The only file we changed was bvh.cpp, specifically the splitting logic inside construct_bvh. The traversal code and everything else stayed the same.

Results

The main win from SAH should be fewer intersection tests per ray since the tree is balanced by expected ray cost instead of just primitive count.

Honestly, the render times came out pretty similar between the two methods. On these particular scenes with 30 threads on the instructional machine, the total render time is dominated by shading and ray generation rather than traversal, so a better BVH split doesn't move the needle much. The intersection tests per ray are already quite low in both cases (single digits), which means even the midpoint heuristic produces a reasonable tree for these meshes. We'd expect SAH to show a bigger advantage on scenes with really uneven geometry distribution or at higher resolutions where traversal cost becomes a larger fraction of total time. The BVH construction itself is still under 20ms for all scenes, so the bucket evaluation overhead is negligible.
